# Step 02 — Set up the Terraform project with git init and .gitignore

## What we were trying to accomplish
Put the project folder under version control (git) BEFORE writing any Terraform,
and add a `.gitignore` so that dangerous/unnecessary files — especially Terraform
state files that can contain secrets — can never be committed.

## Background: what git and .gitignore do
- **git** tracks changes to files over time so you have history, can undo
  mistakes, and (later) collaborate. `git init` makes a folder into a git
  repository by creating a hidden `.git/` subfolder. This is all local until you
  connect it to a remote like GitHub.
- **.gitignore** is a plain-text list of file patterns git should never track.
  Critical for Terraform because Terraform generates files you must not commit.

## What we did, command by command

### 1. Confirm git is installed and the folder isn't already a repo (safe)
```powershell
git --version
if (Test-Path "$PWD\.git") { "Already a git repo" } else { "Not a git repo yet" }
```
- `git --version` → prints git's version (`2.49.0` here).
- The `Test-Path` check → looks for a `.git` folder; printed "Not a git repo yet".

### 2. Initialize the repository
```powershell
git init -b main
```
- `git init` → creates the `.git/` folder, turning this directory into a repo.
- `-b main` → names the default branch `main` (modern convention) instead of the
  older default `master`.
Output: `Initialized empty Git repository in C:/Users/User/Desktop/verafin/.git/`

### 3. Create the .gitignore
Wrote `C:\Users\User\Desktop\verafin\.gitignore`. The most important rules:
- `*.tfstate`, `*.tfstate.*` → **Terraform state files. These often contain
  secrets in plain text. Never commit.**
- `.terraform/` → provider plugin/module cache; large, machine-specific,
  regenerated by `terraform init`.
- `*.tfvars`, `*.tfvars.json` → variable files that often hold secrets
  (with an exception `!*.tfvars.example` so a safe sample CAN be committed).
- `*.tfplan` → saved plan files (may contain sensitive data).
- `crash.log` / `crash.*.log` → Terraform crash logs.
- `override.tf` etc. → local-only experimentation files.
- `.env`, `*.pem`, `.aws/` → belt-and-suspenders blocks for stray secrets.
- OS/editor noise: `.DS_Store`, `Thumbs.db`, `.vscode/`, `.idea/`.

**Deliberately NOT ignored:** `.terraform.lock.hcl`. We *do* commit this file so
provider versions stay pinned and reproducible across machines.

### 4. Verify it works
```powershell
git status
```
Showed `.gitignore` and `docs/` as the only untracked files — exactly what we
want under version control so far.

```powershell
"secret-in-here" | Out-File -Encoding utf8 terraform.tfstate
git check-ignore -v terraform.tfstate
Remove-Item terraform.tfstate
```
- Created a throwaway state file, then `git check-ignore -v` confirmed it is
  blocked by `.gitignore:5:*.tfstate`, then we deleted the test file.
- `git check-ignore -v <file>` is a handy way to ask "is this file ignored, and
  by which rule?" — it prints the file and the matching pattern's line number.

## What went wrong
Nothing went wrong. The repo initialized cleanly and the ignore test passed on
the first try. We have NOT made a first commit yet — that's the next optional
action (requires `git config user.name` / `user.email` to be set).

## Key concepts a beginner should understand
- **Repository:** a folder whose history git tracks, via the hidden `.git/`.
- **Branch:** a line of development; `main` is the default trunk.
- **Tracked vs ignored:** git only versions files it's told to; `.gitignore`
  permanently excludes patterns from being tracked.
- **Why state files are dangerous:** `*.tfstate` can hold secrets in plain text,
  so it must stay out of git (and ideally move to a remote backend later).
- **check-ignore:** `git check-ignore -v <file>` tells you whether and why a file
  is ignored — great for debugging an ignore rule.

## Review questions
1. **Q:** Why is it so important to ignore `*.tfstate` files?
   **A:** Terraform state files record what was built and frequently contain
   secrets in plain text; committing them leaks credentials and causes conflicts.
2. **Q:** Which Terraform file do we intentionally COMMIT, and why?
   **A:** `.terraform.lock.hcl` — it pins exact provider versions so builds are
   reproducible across machines.
3. **Q:** How can you check whether a specific file is being ignored, and by
   which rule?
   **A:** `git check-ignore -v <file>` prints the file plus the `.gitignore`
   line/pattern that matches it (or nothing if it isn't ignored).
